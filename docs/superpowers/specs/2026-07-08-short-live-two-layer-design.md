# 2026-07-08 短线实盘双层推荐设计

## 背景

线上短线正式路径当前是：

- 基础短线：`profile_v9_sector_quality_guard`
- 自适应质量层：`adaptive_quality_v6`
- 强推荐后处理：`SHORT_LIVE_CONSENSUS_PROFILE = "v39"`

v39 的十年回测特点非常明确：胜率和收益质量强，但交易数偏少，部分年份完全没有信号。最近线上也出现过实盘运行成功但短线池为空的情况。用户的核心要求仍然是“买的股票接下来三五天要涨”，因此不能为了补数量把弱信号包装成强推荐。

## 目标

1. 保留 v39 作为唯一“强推荐”来源，不稀释强推荐卡片。
2. 增加一个“观察候选”层，用历史 `best_balance` 思路补覆盖，但明确不等同强推荐。
3. 观察候选最多 Top2，宁少勿滥，避免重新变成杂毛池。
4. 记录强推荐、观察候选、不同入场延迟的 T+1/T+3/T+5 表现，为后续实盘优化提供真实数据。
5. 继续遵守纯选股工具边界，不增加任何自动下单、仓位管理、交易执行代码。

## 非目标

- 不把 `best_balance` 或扩展层直接替换为线上强推荐。
- 不追求每天一定有票。
- 不使用未来数据判断当天应该 T1、T3、T5 买入。
- 不为了提高历史胜率继续细调单年阈值。

## 方案

### 1. 推荐层级

短线输出分两层：

| 层级 | 来源 | 展示位置 | 语义 |
| --- | --- | --- | --- |
| 强推荐 | v39 consensus strong rank guard | 首页强推荐卡片、短线页强推荐卡片 | 可以作为当前短线最强信号观察 |
| 观察候选 | `best_balance` 风格扩展池 | 强推荐卡片下方单独卡片 | 候选跟踪，不标记为强推荐 |

观察候选规则优先采用历史表现更稳的 `best_balance`，而不是收益更激进的 `max_trades`。原因是 `best_balance` 十年总收益为正、交易数约 80、近年胜率较好，并且最差年份回撤明显小于 `max_trades`。

### 2. 观察候选来源

实现上新增配置：

```python
ENABLE_SHORT_LIVE_OBSERVE_LAYER = True
SHORT_LIVE_OBSERVE_PROFILE = "best_balance"
SHORT_LIVE_OBSERVE_TOPN = 2
```

新增或复用策略函数：

```python
build_live_observation_candidates(df, profile="best_balance", top_n=2)
```

候选逻辑只使用当日已可得字段，不能使用未来收益字段。初版规则按历史 `best_balance` 复刻：

- `strong_T1` 信号优先进入观察层。
- 弱动量但板块强共振的扩展信号可进入观察层。
- 震荡市、板块 MA10 强共振的 readiness addon 可进入观察层。
- 排序优先级使用质量、板块共振、资金与综合评分，不做年份特化。

如果某只股票已经进入 v39 强推荐，则从观察候选中去重；页面上只保留强推荐身份。

### 3. 主流程接入

`run_daily_selection` 保持现有强推荐主输出不变：

1. 正常生成短线候选。
2. 先运行 v39 后处理，得到 `stock_pool`，继续作为正式短线强推荐。
3. 在不修改 `stock_pool` 的前提下，对同一批候选额外生成 `observe_pool`。
4. 持久化时给观察层写入独立标识：
   - `profile = "short_live_observe_best_balance"`
   - `source = "live_observe"`
   - `factor_payload.recommendation_layer = "observe"`
   - `factor_payload.observe_profile = "best_balance"`

这样线上正式强推荐和观察候选可以并行回看，不互相污染。

### 4. 页面展示

首页和短线页保留现有强推荐卡片，再新增一张“观察候选”卡片：

- 标题使用“短线观察候选”。
- 文案避免“买入”“强推”等强语义。
- 如果强推荐为空但观察候选有票，页面只显示观察候选，不伪装成强推荐。
- 如果两层都为空，维持现有空态。

### 5. 实盘复盘

新增每日或手动可跑的观察统计，优先复用现有行情缓存和信号库：

| 维度 | 统计 |
| --- | --- |
| T+1 | 买入后第 1 个交易日最大涨幅、收盘收益、最大回撤 |
| T+3 | 3 日窗口最大涨幅、收盘收益、最大回撤 |
| T+5 | 5 日窗口最大涨幅、收盘收益、最大回撤 |
| 延迟入场 | T1、T3、T5 的窗口收益对比 |
| 分层 | strong vs observe |

第一阶段只展示和落库，不让该统计反向影响当天推荐。至少积累 2 到 4 周真实样本后，再判断是否需要把部分观察候选升级成强推荐。

## 风险控制

1. 观察候选最多 2 只，不能无限扩张。
2. 强推荐卡片只认 v39，避免历史扩展层把胜率拉低。
3. 观察候选必须单独 profile/source 入库，避免未来复盘时混淆。
4. 如果观察层连续 4 周真实样本胜率低于 55%，或平均 MAE 明显劣化，则先隐藏观察层。
5. 不引入自动交易相关代码。

## 验证计划

1. 单元测试：验证 v39 强推荐输出不受观察层影响。
2. 单元测试：验证观察候选 TopN、去重、profile/source/factor_payload 标识正确。
3. 服务测试：验证首页和短线页能分别读取 strong 与 observe。
4. 回归测试：跑现有短线相关测试，确认旧卡片和旧接口不破。
5. 手动验证：本地启动 Web 后检查强推荐、观察候选、空态三种页面状态。

## 上线策略

1. 先实现配置和观察层，不改 v39 强推荐。
2. 跑测试并本地检查页面。
3. 合并到 main 后部署服务器。
4. 线上观察 2 到 4 周，记录真实 T+1/T+3/T+5。
5. 再决定是否把观察层中的稳定子类升级为强推荐。

## 推荐决策

本轮建议采用“双层实盘方案”：

- v39 继续做正式强推荐。
- `best_balance` 做观察候选，最多 Top2。
- 先收集真实表现，不直接提高强推荐数量。

这比直接把交易数扩大到 80 更符合当前目标：未来三五天更大概率挣钱，而不是历史表格看起来更热闹。
